Commerce CQRS

One codebase, two highways: commands change state; queries only read it. This page walks through how that split works in this order service.

Back to README See the flows

The idea

Write and read are different jobs

CQRS (Command Query Responsibility Segregation) keeps mutating the domain separate from answering questions. Same SQL database here — different code paths, models, and tools.

Commands → change

Load the Order aggregate, enforce rules, persist with EF Core, raise domain events into the outbox.

CreateOrder ConfirmOrder CancelOrder EF Core

Queries → answer

Skip the aggregate. Run purpose-built SQL with Dapper and return lean DTOs for the UI or API client.

GetOrder SearchOrders Dapper OrderDetails

Write path

Creating an order

POST /api/orders goes through the command handler, the domain, a single SQL transaction, then the outbox worker publishes to Service Bus.

1
API receives the command Minimal API maps JSON to CreateOrder and calls ICommandHandler.
2
Domain builds the aggregate Order.Create validates email, lines, prices — and raises OrderCreated.
3
One transaction commits EF Core saves the order and an outbox row together. No “event without order.”
4
Outbox worker publishes Background service polls pending messages and sends them to Azure Service Bus.

Read path

Fetching an order

GET /api/orders/{id} never hydrates the aggregate. The query handler asks the Dapper read store for a shaped DTO.

1
API receives the query GetOrder is handled by IQueryHandler — no write repository involved.
2
Dapper runs focused SQL SqlOrderReadStore selects order + lines; totals come from SQL.
3
DTO goes back to the client OrderDetails or 404 — optimized for reading, not for enforcing invariants.

Lifecycle

Draft → confirmed or cancelled

Status rules live in the domain. Confirm only from Draft; cancel needs a reason and cannot run twice.

Confirm only from Draft. Cancel needs a reason and works from Draft or Confirmed — never twice. Events OrderCreated, OrderConfirmed, and OrderCancelled leave via the outbox.

Structure

Dependencies point inward

Not event sourcing — SQL is the source of truth. Events are integration signals after a successful commit.

Api
HTTP endpoints, health, OpenTelemetry, ProblemDetails
Application
Commands, queries, handlers, and ports (interfaces)
Domain
Order aggregate, invariants, domain events — no infrastructure
Infrastructure
EF Core writes, Dapper reads, outbox publisher, Service Bus
